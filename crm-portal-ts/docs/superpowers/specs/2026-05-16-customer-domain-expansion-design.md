## Customer Domain Expansion Design

### Goal
- Expand the customer domain from a minimal CRM record into a production-ready operator profile.
- Split customers into two explicit types:
  - `individual`
  - `company`
- Keep one correspondence address on the customer.
- Move installation address responsibility to customer devices, because one customer can own multiple installed services/devices.
- Extend the model with contract and billing fields.

### Recommended Architecture
- Keep a single `customers` table with a `customerType` discriminator.
- Use conditional fields and validation rules based on `customerType`.
- Keep one correspondence address on `customers`.
- Keep installation address on `customer_devices`.
- Keep one main contact for company customers.

This avoids excessive normalization while still supporting the required business rules and a larger data profile.

### Customer Types

#### Individual Customer
- Required:
  - `customerType = individual`
  - `firstName`
  - `lastName`
- Optional:
  - `middleName`
  - `pesel`
  - `birthDate`
  - `idDocumentType`
  - `idDocumentNumber`

#### Company Customer
- Required:
  - `customerType = company`
  - `companyName`
- Optional but supported:
  - `nip`
  - `regon`
  - `krs`
  - `contactFirstName`
  - `contactLastName`
  - `contactPhone`
  - `contactEmail`

### Shared Customer Fields
- `customerCode`
- `status`
- `email`
- `billingEmail`
- `phone`
- `secondaryPhone`
- `notes`
- `portalLogin`
- `portalPasswordHash`
- `lastPortalLogin`
- `isAutoGenerated`
- `autoImportSource`

### Customer Correspondence Address
Customer owns exactly one correspondence address:
- `correspondenceState`
- `correspondenceCounty`
- `correspondenceCity`
- `correspondenceStreet`
- `correspondenceStreetNumber`
- `correspondenceApartmentNumber`
- `correspondencePostalCode`
- `correspondenceCountry`

This address is used as the canonical customer mailing/contact address.

### Customer Contract And Billing
The customer record also stores commercial and billing metadata:
- `contractNumber`
- `contractSignedAt`
- `serviceStartDate`
- `paymentMethod`
- `paymentDueDay`
- `invoiceRecipientName`
- `invoiceRecipientTaxId`
- `billingNotes`
- `marketingConsent`
- `emailConsent`
- `smsConsent`
- `documentDeliveryConsent`

### Customer Device Installation Address
Each `customer_device` can have its own installation address:
- `installationState`
- `installationCounty`
- `installationCity`
- `installationStreet`
- `installationStreetNumber`
- `installationApartmentNumber`
- `installationPostalCode`
- `installationCountry`
- `locationDescription`

This supports one customer with multiple installed devices/services at different physical locations.

### Validation Rules
- `individual`
  - requires `firstName` and `lastName`
- `company`
  - requires `companyName`
- all customer types
  - require `customerCode`
  - respect existing `status` rules
- installation address is optional at DB level but supported in API/UI for each device
- correspondence address is optional at DB level to preserve backward compatibility during migration

### API Changes

#### Customers API
- Extend customer serialization in:
  - list
  - details
  - create
  - update
- Add filtering support for:
  - `customerType`
  - `autoGenerated`

#### Customer Devices API
- Extend create/update/detail payloads with installation address fields.

### UI Changes

#### Customers List
- Add customer type column or badge.
- Keep auto-generated marker visible.
- Add filters:
  - all / individual / company
  - all / auto-generated / normal

#### Customer Form
- Use a conditional form based on `customerType`.
- Sections:
  - basic profile
  - identity/company data
  - correspondence address
  - contract
  - billing
  - consents

#### Customer Device Form
- Add installation address section per device.

### Import And Discovery Compatibility
- Existing fallback customers from network auto-import remain supported.
- Auto-generated fallback customers are created as `individual`.
- They carry minimal customer data and keep `isAutoGenerated = true`.
- Their correspondence address can remain empty until completed manually.
- Imported devices can later receive installation addresses independently.

### Migration Strategy
- Add a new runtime migration extending:
  - `customers`
  - `customer_devices`
- Preserve all existing records.
- Existing customers default to:
  - `customerType = individual`
- Existing devices keep installation fields empty until explicitly filled.

### Testing Strategy
- Add migration coverage for the new fields.
- Add backend tests for:
  - create/update individual customer
  - create/update company customer
  - filtering by `customerType`
  - filtering by `autoGenerated`
  - installation address persistence on customer devices
- Add frontend coverage for customer access/filter rules where appropriate.

### Out Of Scope For This Round
- Multiple company contacts
- Multiple correspondence addresses per customer
- Separate billing address entity
- Fully normalized profile sub-tables for individual/company
